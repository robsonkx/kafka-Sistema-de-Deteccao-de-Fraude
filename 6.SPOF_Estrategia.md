# 1. Camada de Entrada (Edge Layer)

## Componentes

- POS / Clientes
- Load Balancer
- API Gateway

## Estratégia anti-SPOF

### Load Balancer redundante

Em vez de um único balanceador:

Internet
   ─ Load Balancer A
   
   ─ Load Balancer B

Características:

- **active-active**
- health checks
- failover automático

Ferramentas comuns:

- NGINX
- HAProxy
- AWS ALB / GCP Load Balancer

Se um load balancer falhar, o outro continua processando requisições.


### API Gateway escalável

O API Gateway roda **múltiplas instâncias**.

Características:

- stateless
- auto scaling
- health checks

Orquestração normalmente feita com:

- Kubernetes
- Auto Scaling Groups


# 2. Camada de Streaming (Kafka)

Kafka é o **backbone de eventos**, portanto precisa de alta resiliência.

## Cluster Kafka

Kafka opera com múltiplos brokers.

## Replicação de Partições

Cada partição tem múltiplas réplicas.

Exemplo:

Partition 0
   Leader → Broker 1
   Replica → Broker 2
   Replica → Broker 3

Se o líder falhar:

Broker 2 vira leader

## Configurações recomendadas

*replication.factor = 3
min.insync.replicas = 2
acks = all*

Isso garante que os dados **não sejam perdidos mesmo se um broker falhar**.


# 3. Camada de Microserviços

## Fraud Check Service

Os serviços são executados em **múltiplas instâncias**.

## Orquestração com Kubernetes

O Kubernetes garante:

- **auto-healing**
- **auto-scaling**
- **service discovery**

Se um pod falhar:

*Kubernetes cria outro automaticamente*


## Deploy multi-zone

Pods distribuídos em múltiplas zonas:

*Availability Zone A
Availability Zone B
Availability Zone C*

Isso evita falhas por:

- datacenter
- rack
- rede local


# 4. Camada de Cache (Redis)

Redis é usado para **consultas extremamente rápidas**, mas precisa ser resiliente.

## Redis Cluster

Estrutura:

Redis Cluster
 ─ Master 1
      - Replica
 ─ Master 2
      - Replica
 ─ Master 3
       - Replica


## Failover automático

Se um master falhar:

*Replica promove-se automaticamente*

Tecnologias usadas:

- Redis Sentinel
- Redis Cluster


# 5. Camada de Banco de Dados

## Cassandra

Cassandra foi escolhido por ser **distribuído por design**.

### Estrutura

Cassandra Cluster
 ─ Node 1
 ─ Node 2
 ─ Node 3
 ─ Node 4

Dados replicados entre nós.

### Replication Strategy

Exemplo:

*replication_factor = 3*

Cada dado existe em **3 nós diferentes**.

Se um nó falhar:

*outro nó responde a consulta*


## PostgreSQL

Banco relacional precisa de replicação.

Estrutura típica:

PostgreSQL Primary
    ─ Replica 1
    ─ Replica 2

Funções:

Primary:

- escrita

Replicas:

- leitura
- backup

Failover pode ser automático com:

- Patroni
- PgPool
- Aurora / Cloud SQL


# 6. Banco de Grafos (Neo4j)

Neo4j pode operar em **cluster causal**.

Estrutura:

Neo4j Cluster
 ─ Core Node 1
 ─ Core Node 2
 ─ Core Node 3
 ─ Read Replica

Core nodes mantêm consistência.

Se um nó cair:

*cluster continua funcionando*


# 7. Data Lake

O Data Lake é geralmente armazenado em sistemas **distribuídos por natureza**.

Exemplos:

- Amazon S3
- Google Cloud Storage
- HDFS

Características:

- replicação automática
- múltiplas zonas
- alta durabilidade (11 9s)

Portanto, **não há SPOF no armazenamento de dados históricos**.

# 10. Estratégia Multi-Zone / Multi-Region

Para alta disponibilidade real:

## Multi-Zone

Serviços distribuídos em diferentes zonas:

Zone A
Zone B
Zone C

Isso protege contra:

- falhas de rede
- falhas de rack
- falhas de datacenter

## Multi-Region (opcional)

Para sistemas críticos:

*Region 1
Region 2*

Sincronização feita com:

- replicação assíncrona
- backup cross-region

---

Com essa arquitetura:

- nenhuma falha isolada derruba o sistema
- serviços se recuperam automaticamente
- dados permanecem disponíveis
- transações continuam sendo processadas

Isso garante **alta disponibilidade e resiliência**, fundamentais para sistemas financeiros.
