# Estratégia de Coleta de Dados Históricos e Treinamento de Modelos

A estratégia de **Machine Learning para detecção de fraude** deve garantir três objetivos principais:

1. **Coleta confiável de dados históricos**
2. **Construção consistente de features**
3. **Treinamento e atualização contínua dos modelos**

A arquitetura do sistema utiliza **streaming + data lake + feature store** para garantir escalabilidade e qualidade dos dados.

---

# 1. Coleta de Dados Históricos

Os dados históricos são essenciais para treinar modelos capazes de identificar padrões de fraude.

## Fontes de Dados

Os dados são coletados de múltiplos componentes do sistema:

| Fonte             | Tipo de Dados                          |
| ------------------- | ---------------------------------------- |
| Kafka             | Eventos de transação em tempo real   |
| Cassandra         | Histórico operacional de transações |
| PostgreSQL        | Dados de usuários e estabelecimentos  |
| Redis             | listas de fraude e regras              |
| Neo4j             | relações entre entidades             |
| Sistemas externos | listas de fraude globais               |

Esses dados são consolidados no **Data Lake**.


## Pipeline de Coleta de Dados

O pipeline de ingestão funciona de forma **assíncrona e orientada a eventos**.


Fluxo:

POS / API -> Kafka (transactions.raw) -> Stream Processing (Flink / Spark) -> Data Lake (S3 / GCS)


Eventos armazenados incluem:

- transações aprovadas
- transações rejeitadas
- alertas de fraude
- confirmações de fraude
- falsos positivos


## Dados Armazenados

Cada registro de transação contém informações como:

### Informações da Transação

- id da transação
- timestamp
- valor
- moeda
- estabelecimento
- localização

### Informações do Usuário

- id do cliente
- histórico de transações
- país
- perfil de risco

### Informações do Dispositivo

- tipo de dispositivo
- fingerprint
- endereço IP

### Informações Contextuais

- canal da transação (POS, online)
- método de pagamento
- categoria do estabelecimento


## Rotulagem dos Dados (Labeling)

Para treinar modelos de ML, é necessário identificar quais transações são fraudulentas.

Labels são obtidos por:

### Confirmação de fraude

Eventos gerados quando fraude é confirmada:

*fraud.confirmed*


Origem:

- chargeback
- investigação antifraude
- denúncia do cliente


### Falsos positivos

Quando uma transação marcada como fraude é legítima.

Evento:

*fraud.false_positive*

Esse feedback é fundamental para melhorar o modelo.

---

### Transações legítimas

Transações aprovadas sem contestação após um período de segurança.

Exemplo:

transações aprovadas após 60 dias


# 2. Engenharia de Features

As **features** representam padrões de comportamento usados pelo modelo.

Essas features são criadas usando pipelines de streaming.

Tecnologias:

- Apache Flink
- Apache Spark Streaming

---

## Exemplos de Features

### Comportamento do Usuário

*transactions_last_10_minutes

transactions_last_1_hour

transactions_last_24_hours*

---

### Estatísticas de Valor

*avg_transaction_value

max_transaction_value

transaction_std_dev*

---

### Geolocalização

*distance_last_transaction

country_change

velocity_geo*

Exemplo:

compras no Brasil e Europa em poucos minutos.

---

### Comportamento do Dispositivo

*new_device_flag

device_change_frequency*

---

### Padrões de Rede

Dados vindos do **Neo4j**:

*shared_device_count

shared_address_count

fraud_network_score*

---

## Feature Store

As features são armazenadas em uma **Feature Store**, que garante consistência entre treinamento e inferência.

Tecnologias possíveis:

- Feast
  
- Tecton

Funções da Feature Store:

- versionamento de features
  
- reutilização entre modelos
  
- baixa latência para inferência


# 3. Construção do Dataset de Treinamento

Os datasets são construídos a partir do **Data Lake + Feature Store**.

Fluxo:

Data Lake -> Feature Extraction -> Dataset de Treinamento


Cada linha do dataset representa uma transação com:

features + label


Exemplo:

| transação | valor | distância | tx\_1h | device\_change | label     |
| ------------- | ------- | ------------ | ----------- | ------------------- | ----------- |
| T1          | 120   | 5km        | 2         | false             | legítima |
| T2          | 4500  | 2000km     | 12        | true              | fraude    |


# 4. Treinamento do Modelo

O treinamento ocorre em um pipeline automatizado.

Fluxo:

Dataset de treinamento -> Treinamento do modelo -> Validação -> Registro do modelo -> Deploy


Etapas principais:

### 1. Treinamento

O modelo aprende padrões de fraude com base nos dados históricos.

O objetivo é estimar:

*probabilidade de fraude*


### 2. Validação

O modelo é avaliado usando métricas como:

- **Precision**
- **Recall**
- **F1-score**
- **AUC-ROC**

Isso ajuda a balancear:

*fraudes detectadas vs falsos positivos*


### 3. Registro do Modelo

Modelos aprovados são registrados em um **Model Registry**.

Funções:

- versionamento
- 
- controle de rollout
- 
- rollback


# 5. Deploy e Inferência

O modelo é implantado em um serviço de inferência.

Tecnologias:

- TensorFlow Serving
  
- microserviço de scoring

Fluxo:

Transação -> Fraud Check Service -> Feature Store -> ML Inference -> Score de fraude


Resultado:

*score = probabilidade de fraude*


Exemplo:

| Score      | Decisão |
| ------------ | ---------- |
| \< 0.3  | aprovar  |
| 0.3 – 0.7 | revisar  |
| \> 0.7  | bloquear |


# 6. Retraining Contínuo

Fraudes evoluem rapidamente, então os modelos precisam ser atualizados regularmente.

Estratégias:

### Treinamento periódico

*diário ou semanal*

Dados usados:

*últimos 90 dias*


### Aprendizado com feedback

Eventos usados:

*fraud.confirmed

fraud.false_positive*


### Detecção de drift

O sistema monitora:

- mudança no comportamento das transações
  
- mudança na distribuição das features

Se drift for detectado:

*novo treinamento é disparado*


# 7. Monitoramento do Modelo

Após o deploy, o modelo deve ser monitorado continuamente.

Métricas importantes:

### Performance

- precision
- recall
- AUC

---

### Impacto de negócio

- fraude detectada
- taxa de falsos positivos
- perda financeira evitada

---

### Drift de dados

Comparação entre:

*distribuição do treinamento
vs
distribuição em produção*
