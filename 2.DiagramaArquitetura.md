### 1 Diagrama de Arquitetura

A arquitetura é baseada em **processamento de eventos em tempo real**, com separação entre três caminhos principais de processamento:

- **Hot Path** → decisão imediata (\<1s)
- **Warm Path** → processamento analítico e features (minutos)
- **Cold Path** → armazenamento histórico e treinamento de ML

## 1. Origem da Transação (POS → API Gateway)

### Componentes envolvidos

- POS / terminal de pagamento
- Load Balancer
- API Gateway

### Fluxo

1. O cliente realiza uma compra em um **POS**.
2. O POS envia a transação para o sistema usando **HTTPS com mTLS**.
3. A requisição chega ao **Load Balancer**, que distribui para uma instância disponível da API.
4. O **API Gateway** executa:

   - autenticação
   - validação de payload
   - rate limiting
   - auditoria

Após validação, a transação é enviada para o sistema de eventos.

### Saída

A API publica o evento no Kafka:

*transactions.raw*


# 2. Ingestão de Eventos (API Gateway → Kafka)

### Componentes envolvidos

- API Gateway
- Kafka Cluster

### Fluxo

1. O API Gateway serializa o evento da transação.
2. O evento é enviado para um **producer Kafka**.
3. Kafka grava o evento no tópico:

*transactions.raw*

4. O evento é replicado entre **múltiplos brokers** para garantir durabilidade.

### Resultado

A transação agora está disponível para múltiplos consumidores.


# 3. Processamento em Tempo Real (Hot Path)

### Componentes envolvidos

- Kafka
- Fraud Check Service
- Redis

### Fluxo

1. O **Fraud Check Service** consome eventos do tópico:

*transactions.raw*

2. Para cada transação, o serviço consulta o **Redis Cluster**.

Consultas típicas:

*fraud:cards
fraud:users
fraud:sites*

3. O serviço aplica regras de fraude.

Exemplo:

- cartão em blacklist
- usuário com histórico suspeito
- estabelecimento fraudulento

4. O serviço calcula um **score de risco**.

### Decisão

O serviço publica um novo evento:

*transactions.approved*

ou

*fraud.alerts*


# 4. Persistência de Transações

### Componentes envolvidos

- Fraud Service
- Cassandra

### Fluxo

Após a decisão:

1. O serviço grava a transação no **Cassandra**.
2. O Cassandra distribui os dados entre vários nós.
3. A replicação garante tolerância a falhas.

Dados armazenados:

- id da transação
- timestamp
- valor
- status
- score de fraude


# 5. Processamento Analítico (Warm Path)

### Componentes envolvidos

- Kafka
- Flink / Spark
- Feature Store

### Fluxo

1. O pipeline de streaming consome eventos de:

*transactions.raw
transactions.approved*

2. O sistema executa **agregações em tempo real**.

Exemplos:

- número de transações por minuto
- valor médio por usuário
- frequência de uso do cartão

3. As features geradas são armazenadas na **Feature Store**.

Essas features serão usadas por:

- modelos de ML
- análise comportamental


# 6. Armazenamento Histórico (Cold Path)

### Componentes envolvidos

- Kafka
- Data Lake

### Fluxo

1. Um pipeline de ingestão consome dados do Kafka.
2. Os dados são gravados no **Data Lake**.

Tecnologias típicas:

- S3
- GCS
- HDFS

Formato:

*Parquet
Delta Lake*

Dados armazenados:

- eventos de transação
- alertas de fraude
- confirmações de fraude
- falsos positivos


# 7. Treinamento de Machine Learning

### Componentes envolvidos

- Data Lake
- Pipeline de ML
- Model Registry

### Fluxo

1. Dados históricos são extraídos do **Data Lake**.
2. Features são recuperadas da **Feature Store**.
3. Um pipeline de treinamento gera datasets.
4. O modelo é treinado.
5. O modelo é validado.
6. O modelo aprovado é registrado no **Model Registry**.


# 8. Inferência de Machine Learning

### Componentes envolvidos

- Fraud Check Service
- ML Inference Service
- Feature Store

### Fluxo

1. O Fraud Check Service recebe uma transação.
2. O serviço busca features da **Feature Store**.
3. O serviço chama o **ML Inference Service**.
4. O modelo retorna um **score de fraude**.

Exemplo:

| Score     | Decisão |
| ----------- | ---------- |
| \< 0.3 | aprovar  |
| 0.3–0.7  | revisar  |
| \> 0.7 | bloquear |


# 9. Detecção de Redes de Fraude

### Componentes envolvidos

- Cassandra
- Neo4j

### Fluxo

1. Transações e entidades são exportadas para o **Neo4j**.
2. O banco de grafos cria relações entre:

- usuários
- dispositivos
- cartões
- estabelecimentos

3. Algoritmos de grafos identificam padrões suspeitos.

Exemplos:

- múltiplos cartões no mesmo dispositivo
- contas conectadas a um mesmo endereço

# 10. Monitoramento e Observabilidade

### Componentes envolvidos

- Prometheus
- Grafana
- OpenTelemetry
- ELK

### Fluxo

1. Cada serviço expõe métricas.
2. O **Prometheus coleta métricas**.
3. O **Grafana exibe dashboards**.
4. Logs são enviados para o **ELK Stack**.
5. Traces distribuídos são coletados com **OpenTelemetry**.

Métricas monitoradas:

Infraestrutura

- CPU
- memória
- latência de serviços
- consumer lag Kafka

Negócio

- taxa de fraude
- falsos positivos
- transações por segundo
