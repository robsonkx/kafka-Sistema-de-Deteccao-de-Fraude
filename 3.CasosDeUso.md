# Caso de Uso 1 — Processamento de Transação Legítima

## Atores

- Usuário (cliente do cartão)
- POS / terminal de pagamento
- Sistema de detecção de fraude
- Banco emissor

## Descrição

Uma transação é iniciada por um cliente e precisa ser validada rapidamente para garantir uma boa experiência de pagamento.

## Fluxo principal

1. O cliente realiza uma compra utilizando o cartão em um **POS**.
2. O POS envia a transação para a **API Gateway** via conexão segura (mTLS).
3. A requisição passa pelo **Load Balancer**, que distribui a carga entre instâncias da API.
4. A API valida o payload e publica o evento no **Kafka (transactions.raw)** .
5. O **Fraud Check Service** consome o evento.
6. O serviço consulta o **Redis Cluster** para verificar se:

   - cartão está em blacklist
   - usuário tem histórico fraudulento
   - estabelecimento é suspeito
7. Nenhuma regra de fraude é encontrada.
8. A transação é **aprovada**.
9. A transação é registrada no **Cassandra**.
10. O evento é enviado para **transactions.validated** no Kafka.
11. O POS recebe confirmação da aprovação.

## Resultado

A transação é aprovada em **menos de 1 segundo**.


# Caso de Uso 2 — Detecção de Fraude Conhecida

## Atores

- Usuário fraudador
- Sistema de detecção de fraude
- Sistema antifraude

## Descrição

Uma transação envolve um cartão ou conta já identificada anteriormente como fraudulenta.

## Fluxo principal

1. A transação chega ao sistema pelo **API Gateway**.
2. O evento é publicado no **Kafka**.
3. O **Fraud Check Service** consome o evento.
4. O serviço consulta o **Redis**.
5. O Redis retorna um registro indicando que o cartão está listado em:

*fraud:cards*

6. O serviço classifica a transação como fraudulenta.
7. Um evento é publicado no tópico:

*fraud.alerts*

8. A transação é **bloqueada imediatamente**.
9. O sistema envia notificação ao banco emissor.

## Resultado

- transação rejeitada
- alerta de fraude registrado
- evento enviado para investigação


# Caso de Uso 3 — Detecção de Fraude Comportamental

## Atores

- Sistema antifraude
- Motor de machine learning

## Descrição

Uma transação inicialmente parece legítima, mas padrões de comportamento indicam fraude.

## Exemplo

- cartão usado em dois países diferentes em poucos minutos
- valor muito acima do padrão do usuário
- muitas transações seguidas

## Fluxo

1. A transação é recebida e **aprovada no Hot Path**.
2. O evento é consumido pelo pipeline de streaming (**Flink/Spark**).
3. O sistema calcula features comportamentais:

Exemplo:

- número de transações na última hora
- valor médio por transação
- distância geográfica da última compra

4. As features são armazenadas na **Feature Store**.
5. O modelo de ML executa inferência.
6. O modelo classifica a transação como suspeita.
7. Um evento é publicado no Kafka:

*fraud.alerts*

8. O sistema antifraude notifica analistas ou bloqueia o cartão.

## Resultado

- fraude detectada **pouco tempo após a transação**
- possível bloqueio do cartão
- alerta para investigação


# Caso de Uso 4 — Identificação de Redes de Fraude

## Atores

- Sistema antifraude
- Analistas de fraude

## Descrição

Fraudadores muitas vezes operam em **redes organizadas**. O sistema identifica conexões entre contas.

## Fluxo

1. Transações e entidades são armazenadas no **Neo4j**.
2. O sistema constrói grafos conectando:

   - cartões
   - usuários
   - dispositivos
   - estabelecimentos
3. Algoritmos de grafos são executados:

Exemplos:

- detecção de comunidades
- análise de centralidade
- detecção de ciclos

4. O sistema identifica padrões como:

- múltiplos cartões usando o mesmo dispositivo
- várias contas conectadas a um mesmo endereço
- padrões circulares de transações

5. O sistema gera alertas adicionais.

## Resultado

Descoberta de **fraudes organizadas**.


# Caso de Uso 5 — Falso Positivo

## Atores

- Cliente
- Analista de fraude
- Sistema antifraude

## Descrição

O sistema marca uma transação como fraudulenta, mas ela é legítima.

## Fluxo

1. O sistema gera um alerta de fraude.
2. O cliente contesta a transação ou o analista revisa o alerta.
3. O analista confirma que a transação é legítima.
4. Um evento é registrado:

*fraud.false_positive*

5. O evento é enviado ao **Data Lake**.
6. O evento é utilizado no próximo ciclo de treinamento do modelo.

## Resultado

- redução futura de falsos positivos
- melhoria do modelo


# Caso de Uso 6 — Treinamento do Modelo de Machine Learning

## Atores

- Pipeline de ML
- Cientistas de dados

## Descrição

O sistema utiliza dados históricos para melhorar a detecção de fraude.

## Fluxo

1. Eventos de transação são armazenados no **Data Lake**.
2. Dados confirmados de fraude são coletados de:

*fraud.confirmed
fraud.false_positive*

3. O pipeline de dados cria datasets de treinamento.
4. Features são extraídas da **Feature Store**.
5. O modelo é treinado.
6. O modelo é validado e registrado.
7. O novo modelo é implantado no serviço de inferência.

## Resultado

O sistema melhora continuamente sua capacidade de detectar fraudes.
